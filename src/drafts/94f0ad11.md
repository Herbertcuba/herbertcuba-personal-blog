Every time your AI agent misses something, you add a rule. A skill. A paragraph in the system prompt. The fix feels surgical in the moment. Six months later, you've got 27,000 characters of skill descriptions loading before the user types a single word.

Nate B. Jones ran a harness audit recently and found exactly that. An agent configured with 27,000 characters of skill descriptions against an 8,000-character discovery budget. The model could not read its own configuration. The instructions meant to help it were too long for it to find.

I've done this. You fix a failure once, you write a rule so it never happens again, and you move on. The rule stays. It always stays. And every new rule makes the context window a little more crowded, a little more diluted. The model isn't getting dumber. You're suffocating it.

## The entropy you can't see

The Nostromo in Alien is a mining ship. The crew trusts it. They know its systems, its sounds, its quirks. What they don't know is that something has rewritten the ship's protocols without their knowledge. The systems look the same. The overrides are different. By the time the crew notices, the ship is working against them.

That's what harness entropy looks like. Your agent setup is the Nostromo. Every rule you added was reasonable. Every skill solved a real problem. But the accumulation changed the ship's behavior in ways you never tested for, because you tested each addition in isolation and never retested the whole.

Jones ran a comparison. A compact harness with only essential skills finished 3 out of 3 tasks. The bloated version broke delivery requirements twice. Same model. Same tasks. The only variable was how much instruction overhead the model was carrying.

## The harness is engineering, not configuration

A Hacker News thread last week pushed this further. "The agent harness belongs outside the sandbox," 182 points. The argument: your harness is a separate engineering concern, not a config file you append to. It needs version control, testing, pruning. The same discipline you'd apply to any codebase that grows over time.

Ameya Gholap has been doing something interesting on this front. He uses a CLAUDE.md pattern with prompt-capture-with-indexes, so every instruction that goes into the harness is traceable. You can see what's loaded, when it was added, and why. I don't know many teams doing this.

Most people treat the harness as a living document that only grows. It should be a codebase that gets refactored.

## This is an operating model problem

I wrote about harness as product a few weeks ago. This is the other side. The harness isn't just your product. It's your technical debt. And the debt compounds silently.

When a model upgrade ships and your agent suddenly degrades, you blame the model. But the model didn't change your harness assumptions. Your harness assumed a context window size, a reasoning style, a tool-calling format. The upgrade shifted one of those, and the 18,000 words of instructions you accumulated over six months broke in ways you can't diagnose because you never documented what each rule was for.

The teams that own their harness like engineering discipline will pull ahead. The ones that let it grow barnacles will keep blaming the model for degradation they caused. Your harness is your competitive edge, or it's your anchor. Same code. Different maintenance.
