A few months ago I found myself running ten agents across three different projects. Code agents, content agents, research agents. They were producing work. Some of it was good. Some of it was quietly terrible, and I didn't catch it until it had already shipped downstream. The problem wasn't the agents. The problem was me. I had no system for how humans and agents should work together. No structure for who decides what, when agents should iterate on their own, and when a human needs to step in.

I looked around for frameworks. There's no shortage of writing about how to work with AI agents. Most of it is good. And most of it shares the same blind spot: it teaches one person how to work with agents. It doesn't show how to scale that into a team of humans. When you have three, five, ten people who all need to interact with an agentic system, the individual playbook breaks down. You need organizational structure: who owns what, how quality is governed, how the system evolves.

That's the gap APX fills. Not another guide for how an individual should prompt or configure an agent. A framework for how organizations plug agentic production into their existing teams.

This document is the complete reference. Twelve chapters covering everything from core beliefs to anti-patterns to adoption paths. Use the index below to navigate.

## Chapter 1: Introduction

*APX exists because individual AI playbooks don't scale to teams. This chapter defines the problem, the framework's scope, and the APX Cycle that holds it all together.*

### The problem APX solves

The gap between "one person using agents well" and "a team of humans running agentic production" is where most organizations stall. Individual productivity with agents is a solved problem at this point. Organizational productivity with agents is not. You need structure for who owns what, how quality is governed, how the system evolves. That structure is what APX provides.

### What APX is (and is not)

APX stands for Agentic Production Execution. It is a framework for how organizations plug agentic production into their existing teams. It is not a guide for how an individual should prompt or configure an agent. It is not a replacement for your existing methodology. It is organizational scaffolding for human-agent collaboration at scale.

I mapped the 15 capabilities in [The Anatomy of a Level 3 Agent](/posts/2026-03-27-anatomy-level-3-agent/). That gave me a vocabulary for what agents can do. But capabilities don't tell you who owns quality, how work flows from idea to verified deliverable, or what happens when the system itself needs to change. APX answers those questions.

### The APX Cycle

The structure is a APX Cycle with three phases. **Human-First**, where humans design the system before agents touch anything. **Agentic-First**, where agents execute and iterate within that designed system. **Human-Last**, where humans verify the output and close the loop. Then you calibrate and the loop starts over, but now the system is smarter than it was last cycle.

Human-First → Agentic-First → Human-Last → Calibrate → repeat. That's the rhythm. Each phase has its own responsibilities, its own artifacts, its own failure modes.

![The APX Cycle](/images/posts/apx-meta-loop.svg)

Human-First is the strategic phase. Everything that happens before agents start executing. If you've ever run a discovery phase, this will feel familiar: defining requirements, documenting context, setting up tooling, aligning stakeholders. The difference is speed. AI accelerates this phase significantly, but every output still needs human sign-off. This is Level 1 or Level 2 acceleration, useful but not where the structural shift happens. That comes in Phase 2, when the APX Loop takes over and pushes you toward Level 3.

In my experience, Human-First is where most agentic projects either succeed or fail, and most people want to skip past it because it feels like overhead. It is not overhead. It is the work.

## Chapter 2: Core Beliefs

*APX rests on seven convictions arrived at through running agentic teams daily. Not theory. Practice.*

![The Seven Beliefs](/images/posts/apx-beliefs.svg)

**1. Agents are team members, not tools.** If you treat an agent like a fancy script, you'll configure it like a fancy script, and the output will reflect that. Agents need roles, responsibilities, memory, and communication protocols, the same way any team member does. The difference is that you have to be explicit about all of it because an agent won't infer organizational culture from the hallway.

**2. Humans specify and verify. Agents execute and iterate.** Humans sit at the bookends of the process. Not in the middle reviewing every small decision, not micromanaging iterations, but at the start designing the system and at the end verifying the output. APX moves humans from trivial decisions to strategic decisions, and if you do it right, the humans on your team are more engaged, not less.

**3. Static configurations degrade.** No system stays good forever. Requirements shift. Agent capabilities improve. Competitors move. The system has to evolve, which means you need a mechanism for evolution baked in from the start. That mechanism is calibration through data-driven reflections.

**4. Trust is earned through architecture, not faith.** You don't trust an agent because it seemed reliable last week. You trust it because you've defined its permissions, set its guardrails, and established quality gates that catch failures before they reach users. Architecture, not hope.

**5. Output quality is a function of input quality.** An agent working from a vague one-liner will produce vague output. An agent working from a detailed spec with brand guidelines, technical constraints, and clear acceptance criteria will produce something you can actually use. Garbage in, garbage out has been true since the 1950s. It's still true.

**6. Outsource execution, keep strategy human.** The temptation with powerful agents is to let them figure out strategy too. Don't. The moment you let agents decide what to build rather than how to build it, you've lost the thread.

**7. The harness determines the ceiling.** Your choice of runtime determines everything else. A general-purpose agent in a chat window has different constraints than a specialized DAG pipeline. Choose it deliberately.

### What APX is not

APX is not a replacement for product thinking. It assumes you already know what to build and why. It is not a prompt engineering guide. It operates one level above individual agent configuration. It is not prescriptive about tooling. The five domains and the APX Cycle work regardless of which LLM provider, orchestration framework, or deployment platform you use.

## Chapter 3: The Five Domains

*APX organizes Human-First work into five domains, each producing specific artifacts that agents consume during execution. Getting the boundaries right prevents the overlap and gaps that kill agentic operations.*

![The Five Domains](/images/posts/diagram-six-domains.svg)



Each domain produces specific artifacts that agents consume during execution. Each domain also has clear boundaries: what it owns, what it doesn't, and where it interfaces with adjacent domains. Getting these boundaries right prevents the overlap and gaps that kill agentic operations.

I've seen a content operation skip Trust Architecture entirely because "it's just blog posts." Three weeks later an agent published a draft with unverified statistics to a live site. The domain existed whether they acknowledged it or not. If you skip a domain, you create a blind spot that agents will fill with assumptions, and agent assumptions are expensive.

### 3.1 Platform Architecture

Platform Architecture is the foundation. It covers two things that are inseparable in practice: the harness decision (which runtime you use) and the infrastructure that keeps it running.

The harness decision comes first. This is where you choose your runtime: general purpose, specialized, autonomous, hierarchical, DAG-based, or hybrid. The choice constrains every other domain — what orchestration patterns are possible in Workflow Design, what deployment model Quality Engineering can rely on, what security boundaries Trust Architecture can enforce. Document it in a Harness Decision Record.

Then comes the model strategy. Which LLMs power which agents? Do you need local models for latency, privacy, or cost reasons, and if so, what infrastructure supports that? What are your fallback models when a primary provider hits rate limits or goes down? These decisions are tightly coupled to the harness — some runtimes support multi-model routing and fallbacks natively, others don't. A content agent might run on a cheaper, faster model while a code review agent needs a reasoning-heavy one. Document the model assignments per agent and the fallback chain.

Beyond models, the system needs to be stable, properly resourced, and maintained. Platform Architecture owns the infrastructure: cloud or on-premise compute, CI/CD pipelines, tool integrations, monitoring, and agent deployment. When a harness needs updating, when an integration breaks, when a model provider changes pricing or capabilities — that's Platform Architecture. The artifacts are a Harness Decision Record, a Model Strategy (assignments + fallbacks), a Tool Registry, and an Infrastructure Config.

Platform Architecture does not own what agents do within the runtime (that's Workflow Design) or what context they receive (that's Intent Architecture). It owns the container — the shape, the stability, and the tooling. The other domains fill it.

### 3.2 Intent Architecture

Intent Architecture is where most of the investment goes. This is context, memory, and state. You collect client briefs, project briefs, brand guidelines, design systems, technical documentation, compliance requirements. You produce spec documents and memory logs. In a content operation, this means editorial guidelines. In a financial operation, risk parameters and regulatory constraints. In a legal operation, case law references and precedent databases.

Intent Architecture owns everything an agent needs to know to produce domain-appropriate output. It does not own how agents are orchestrated (Workflow Design) or what tools they access (Platform Architecture). The boundary with Quality Engineering is worth noting: Intent Architecture defines what "good" means in the spec. Quality Engineering defines how you verify that "good" was achieved. Input side meets output side through acceptance criteria.

**Intent Architecture: a best-practice guide**

Intent Architecture deserves special attention because it produces the richest artifact collection and has the most direct impact on agent output quality. The relationship is almost linear: richer Intent Architecture produces better agent output.

The work itself is familiar. It's the same discovery and documentation work that product teams have always done. The difference is that these artifacts now serve a dual audience: humans for alignment, agents for operational context. That dual purpose means they need to be more explicit and more complete than what most teams are used to producing. Ambiguity that a human colleague would resolve by walking over to your desk becomes a source of drift when an agent encounters it.

I think of Intent Architecture as a phased build. Each phase layers on the previous one, and the artifacts accumulate into progressively richer context.

**Phase 1: Foundation.** This is the business case, the brand understanding, the competitive context, and the high-level requirements. Why does this product or project exist? Who is it for at a macro level? What does the market look like? What are the top-line capabilities framed as business outcomes? These documents give agents the strategic framing they need to make sensible decisions within their scope. Without foundation documents, agents produce work that's technically correct but strategically disconnected. I've worked on a project where a content agent was generating excellent copy that completely missed the competitive positioning because nobody had documented what made the product different. The foundation phase prevents that.

**Phase 2: Users and experience.** Personas with goals, pain points, and behaviors. User journey maps from awareness through advocacy. Product-specific brand guidelines for visual identity, tone, and accessibility. Information architecture defining structure and navigation. These artifacts teach agents who they're building for. A code agent working from journey maps writes better error handling. A content agent working from personas writes for real people instead of generic audiences.

**Phase 3: Specification.** The product requirements document, user stories with acceptance criteria, MVP scope and priorities, content guidelines. This is the translation layer between strategic thinking and executable work. The PRD becomes the single source of truth. User stories with clear acceptance criteria are what agents decompose into tasks. Phase 3 is the handoff between product thinking and agentic execution. If the PRD is thin, every agent downstream compensates with assumptions.

**Phase 4 and beyond: Technical and launch.** Technical architecture, API contracts, integration specs, QA strategy, and launch readiness. These artifacts serve the agents doing the actual building and the humans doing the actual verification. As the project matures, these documents become the densest part of the Intent Architecture.

The phased approach prevents the common failure of trying to document everything upfront. You don't need API contracts in week one. You need a business case and high-level requirements. Each phase builds on the previous one and produces artifacts that are immediately useful, even if later phases haven't started.

One practical observation: I've seen teams treat Intent Architecture as a one-time exercise. Write the docs, hand them to agents, move on. That's the Set-and-Forget anti-pattern applied to documentation. Intent Architecture is a living collection. As agents execute and you learn from reflections, the artifacts evolve. Specs get more precise where ambiguity caused extra iterations. Brand guidelines get more detailed where agents drifted from the intended tone. The phased build never really ends. It shifts from initial creation to iterative refinement.

### 3.3 Workflow Design

Workflow Design owns the agentic flows and the agents themselves. This is where you define how agents take in context from Intent Architecture and produce the output you need. Agent identity files, agent behavior configuration, orchestration patterns, delegation chains, routing rules — all of it lives here. The artifacts are an Agent Roster, Agent Identity Files, and a Workflow Map.

This domain is about performance in the truest sense: how well do your agents consume context and produce the right output? If an agent consistently misinterprets a spec, that's a Workflow Design problem — the agent's identity, instructions, or routing need adjustment. If the handoff between two agents drops information, that's a Workflow Design problem. If output quality drifts because an agent's behavior wasn't calibrated, that's a Workflow Design problem.

Workflow Design does not own the content of the work (Intent Architecture) or the infrastructure (Platform Architecture). It interfaces upstream with Platform Architecture, which constrains what orchestration patterns are possible, and downstream with Quality Engineering, which defines what checks happen at each handoff.

### 3.4 Quality Engineering

Quality Engineering defines validation criteria and produces Review Criteria documents and Reflection Reports.

Quality Engineering owns how quality is measured, what checks run at each stage, and how reflection data is collected. It does not own the acceptance criteria themselves, which come from Intent Architecture and domain experts. Quality Engineering turns those criteria into automated and semi-automated checks that agents execute. It interfaces with every other domain, but the primary handshake is with Intent Architecture on input and Workflow Design on placement.

### 3.5 Trust Architecture

Trust Architecture covers security, permissions, and compliance, producing a Permission Map and a Domain Ownership Matrix.

Trust Architecture owns who can access what, what data flows where, and what regulatory constraints apply. It does not own technical infrastructure decisions (Platform Architecture) or work quality definitions (Quality Engineering). It interfaces most heavily with Platform Architecture, which implements the permission boundaries, and with Intent Architecture, which may include compliance requirements in agent context documents. In regulated industries, Trust Architecture often becomes the constraining factor across every other domain.

### 3.6 The Inner Loop vs the Outer Loop

Humans don't operate inside the execution loop. The inner loops belong to agents: spec, execute, review, iterate. Humans operate in the outermost loop: designing the system, setting constraints, and verifying results. The five domains are where human expertise translates into that outer loop.

A tech lead who used to review every pull request now defines quality criteria and review rules that agents follow autonomously. A product manager who used to write individual user stories now writes specs that agents decompose and execute. A content strategist who used to edit every article now defines editorial guidelines that review agents enforce. The expertise doesn't disappear. It moves up one level of abstraction, from doing to orchestrating.

For some experienced leaders this is second nature. They already think in systems. For others it takes repetition and practice. Both are fine. The important part is recognizing that the translation from domain expertise to systems thinking is the core skill APX asks of humans.

## Chapter 4: Domain Ownership

*Not everyone needs to own every domain. In fact, that's an anti-pattern. Domain ownership maps to expertise, and the mapping changes as your organization evolves.*


### 4.1 How Domains Map to Roles

Domain ownership maps to expertise. The person who understands the runtime tradeoffs owns Harness Architecture. The person closest to the business problem owns Intent Architecture. The person who thinks in systems and flows owns Workflow Design. Matching domain to expertise prevents the scenario where someone is technically responsible for a domain they don't understand well enough to configure correctly.

### 4.2 Role Examples per Domain

**Harness Architecture** aligns with Tech Directors, Tech Leads, AI Engineers, Solutions Architects. The people who understand runtime tradeoffs.

**Intent Architecture** aligns with Product Managers, Design Leads, Business Analysts, Content Strategists. The people closest to the business problem and the users.

**Workflow Design** aligns with Product Managers, Tech Leads, Process Designers, Scrum Masters. The people who think in systems and flows.

**Platform Architecture** aligns with DevOps Engineers, AI Engineers, Infrastructure Engineers. The hands-on operators.

**Quality Engineering** aligns with Tech Directors, QA Leads, Product Managers. The people who define "good enough" and design the review criteria.

**Trust Architecture** aligns with Security Engineers, Compliance Officers, Legal. The people who understand organizational risk tolerance and regulatory requirements.

### 4.3 Small Team vs Large Organization

A single person can own multiple domains, especially in smaller teams. In a two-person operation, one person might own Intent Architecture, Workflow Design, and Quality Engineering while the other owns Platform Architecture, Platform Architecture, and Trust Architecture. In a larger organization, each domain has a dedicated owner.

The Domain Ownership Matrix is a living document. Review ownership at every reflection. Ask: does this person still have the right expertise and bandwidth? Sometimes the answer changes.

## Chapter 5: The Ten Principles

*These principles govern how APX operates across all three phases. Each one earns its place through practice, not theory.*

![Ten Principles](/images/posts/apx-principles.svg)

**Principle 1: Harness First.** Your runtime choice sets all constraints. A hierarchical multi-agent system has different capabilities, costs, and failure modes than a single autonomous agent. Decide the harness before you configure anything else.

**Principle 2: Human at the Bookends.** Humans specify and verify. Agents execute and iterate in the middle. This is the core labor division of APX.

**Principle 3: Spec Quality = Output Quality.** Agent output is never better than the spec it's working from. If you want better output, write better specs. This sounds obvious. In practice, it's the most common point of failure.

**Principle 4: Agents Review Agents First.** All work passes through agent-to-agent review before a human sees it. This is how you get speed without sacrificing quality.

**Principle 5: Domain-Mapped Ownership.** Quality gates map to expertise, not generic reviews. The person who understands the code reviews the code. The person who understands the brand reviews the content. No one reviews everything.

**Principle 6: Iterate Often, Iterate Fast.** Agent-to-agent iteration loops compress what used to take days into hours. Use that speed. Don't gate iterations behind human approval when agents can resolve the issues themselves.

**Principle 7: Least Privilege.** Every agent gets only the access it needs. No more. A content agent doesn't need production database credentials. A code agent doesn't need access to financial records.

**Principle 8: Calibrate the System, Not Just the Output.** Reflections improve the system itself, not just the current deliverable. If you find yourself fixing the same kind of issue repeatedly, the system needs to change.

**Principle 9: Data-Driven Reflections.** Agents report metrics. Iteration counts, error rates, review pass rates, time-to-completion. Humans decide based on data, not gut feelings about whether things are "going well."

**Principle 10: Think Big, Scale Back.** Start with the vision of maximum efficiency, the full system running at scale. Then scale back what doesn't work yet. The opposite approach, cautiously adding one AI tool to one workflow, produces a copilot. Not the structural shift. Design the whole system first. Remove what's premature. Keep the architecture.

## Chapter 6: APX Metrics

*Principles without measurement are just opinions. These five metrics tell you whether the system is improving or degrading, and they make data-driven reflections possible.*

![APX Metrics Dashboard](/images/posts/apx-metrics.svg)

APX needs its own equivalent of DORA metrics, a small set of named indicators that cover the full loop. Without these, data-driven reflections are impossible. You're just guessing. I've landed on five.

**First-Pass Acceptance Rate** is the percentage of deliverables accepted at Human-Last verification without being sent back for another iteration. This is the clearest signal of spec quality. If the rate is high, your Intent Architecture is doing its job. If it's low, something upstream is broken: vague specs, missing context, unclear acceptance criteria. Track this per domain and per deliverable type. A high first-pass rate on code but low on content tells you exactly where to invest.

**Iteration Depth** is the average number of agent-to-agent iterations per task before it reaches human verification. This measures both spec quality and agent capability. The interesting signal is the trend. If iteration depth decreases cycle over cycle, your specs are getting sharper. If it increases, something is drifting. High iteration depth with high first-pass acceptance means agents are catching and fixing issues before humans see them, which is the system working as designed. High iteration depth with low first-pass acceptance means something more fundamental is wrong.

**Human Touch Rate** is the percentage of tasks that require human intervention during the Agentic-First phase, outside of the designed verification points. Every time a human steps into the agent loop to unblock or clarify something, that's a touch. This should decrease over time. A high touch rate in early cycles is normal. A high touch rate after ten cycles means your Human-First configuration has gaps that calibration isn't closing. The goal isn't zero. The goal is that the touches are intentional, not symptoms of poor system design.

**Calibration Impact** measures the change in the other metrics from one cycle to the next. It's the meta-metric. It tells you whether reflections are producing meaningful system improvements or whether you're going through the motions. If Calibration Impact is flat across cycles, the ceremony is happening but the learning isn't.

**Cycle Velocity** is the time from spec to verified delivery. As the system matures, this compresses. Better specs mean fewer iterations. Better agents mean faster execution. Better quality gates mean faster verification. Track per deliverable type because a complex feature and a blog post have fundamentally different velocity profiles.

Together these five metrics make reflections concrete. Instead of "I think things are going better," you say "First-pass acceptance improved from 60% to 78%, iteration depth dropped from 4.2 to 3.1, and human touch rate is down to 12%." That's a conversation you can act on.

## Chapter 7: The APX Loop

*The core execution cycle where agents do the actual work. This is the Agentic-First phase, and it should run with minimal human intervention.*

![The APX Loop](/images/posts/apx-loop-detailed.svg)

### 7.1 Spec → Execute → Review → Iterate → Verify

The system is designed. The artifacts are in place. Agents execute within the boundaries that Human-First established. The key insight: this phase should run with minimal human intervention. If you find yourself constantly stepping in during Agentic-First, the problem is in your Human-First configuration, not in the agents.

The core execution cycle is straightforward. A human writes a spec. An agent executes against that spec. Another agent reviews the output. Issues are identified and the executing agent iterates. This continues until the spec is met, at which point a human verifies the final output.

Spec → Execute → Review → Iterate → Verify.

This loop is domain-agnostic. In a product development context, the spec might be a feature brief with acceptance criteria. The executing agent writes code. A review agent checks it against the criteria, runs tests, flags issues. The executing agent fixes them. After two or three iterations, the output meets the spec and goes to human verification. The human never saw the intermediate versions. They see the polished result.

In a content operation, the spec might be an editorial brief with tone guidelines and SEO requirements. The executing agent writes the article. A review agent checks it against the brief, flags sections that drift from the brand voice, identifies missing keywords. The executing agent revises. Same loop, different domain. Same loop for financial analysis, legal research, or any other context where work can be specified and verified.

The loop runs N times per work cycle. Sometimes that's three iterations for a simple task. Sometimes it's fifteen for a complex deliverable. The number of iterations is not a problem. Slow iterations are a problem. Agent-to-agent iterations are fast because agents don't need coffee breaks, don't get pulled into meetings, and don't have ego about their first draft.

### 7.2 Agent-to-Agent Review

Principle 4 says all work passes through agent-to-agent review before a human sees it. This is counterintuitive for people used to traditional review flows where a senior person reviews junior work. In APX, the first line of quality assurance is another agent.

Agent review is cheap and fast. A review agent checks work against documented criteria in seconds. It catches spec mismatches, style violations, missing sections, logical inconsistencies. Humans are expensive and slow at catching those kinds of issues.

Agents catch the mechanical issues. Humans catch the strategic issues: taste, business context, user needs, organizational politics, the soft knowledge that can't be captured in a spec document. By the time a human sees the work, the obvious problems are resolved. The human focuses on "Is this the right thing?" rather than "Is this thing done correctly?"

## Chapter 8: Calibration Cadence

*The rhythm that keeps the system alive. Without calibration, APX is a static pipeline. With it, APX is an evolving system.*

![Calibration Cadence](/images/posts/apx-calibration.svg)

### 8.1 Verification → Reflection → Calibration

The Calibration Cadence follows a predictable pattern: work cycle, then verification, then reflection, then calibration, then the next work cycle begins.

A work cycle can be whatever fits your operation. A day, a week, a two-week period, an entire project. The cadence doesn't prescribe duration. What it prescribes is that every work cycle ends with three things.

A **verification** where the human verifies the cycle's output against the original intent. Not a status meeting. The actual output, live.

A **reflection** where both humans and agents brainstorm what could be tweaked and enhanced for better results. Agents report their APX Metrics: first-pass acceptance rate, iteration depth, human touch rate. Humans review the data and identify patterns.

A **calibration** where the team implements the changes they believe will make impact, adjusting the strategic layer for better outcomes in the next cycle. Maybe the Intent Architecture needs more detail because agents keep misinterpreting a requirement. Maybe workflow routing is inefficient. Maybe permissions need tightening. These changes flow back into Human-First, and the next work cycle runs on the evolved system.

This is what separates APX from a static pipeline. A pipeline runs the same way forever. APX evolves.

### 8.2 Adoption Sequence

You don't need to build all five domains at full depth before you start. The adoption sequence depends on what you're doing.

If your primary work is **product development**, invest heaviest in Intent Architecture first. Code agents live or die on spec quality. Get your feature briefs, technical documentation, and acceptance criteria right, and the rest follows.

If your primary work is **content and editorial**, invest heaviest in Harness Architecture first. Content operations need the right orchestration model because the workflow between research, writing, editing, and publishing is complex and varies by content type.

If your primary work is **financial analysis or compliance**, invest heaviest in Trust Architecture first. Permissions, audit trails, and regulatory constraints have to be airtight before agents touch anything sensitive.

Start with the domain that matters most. Get it working. Add depth to the others as you learn what's needed through calibration. Trying to fully configure all six before running your first work cycle is its own anti-pattern. Configure enough to start, then let the reflection data tell you where to invest next.

## Chapter 9: Human-Last Verification

*The verification phase where humans close the loop. By the time work arrives here, the mechanical issues are resolved. What remains are the strategic calls that require human judgment.*


### 9.1 Quality Gates Mapped to Expertise

The biggest mistake in verification is treating it like a single checkpoint. "Someone senior reviews everything before it ships." That approach doesn't work.

In APX, quality gates map to domain expertise. The person who understands the code reviews the code artifacts. The person who understands the brand reviews the content. The person who understands the regulatory environment reviews compliance. Each quality gate has a specific owner, documented in the Domain Ownership Matrix.

This means verification is parallel, not serial. While one person verifies technical requirements, another verifies brand voice. They're not waiting for each other. In my experience, the review bottleneck is the single biggest drag on delivery speed, and it exists because organizations default to serial, generalist review. Domain-mapped quality gates fix that.

### 9.2 Final Delivery Verification

Verification ends with acceptance, not a rubber stamp. The person responsible sees the actual output in its actual context. A working demonstration, a formatted article, a report with real data. It is a binary decision: accepted or not accepted. If not accepted, specific feedback is captured and the work re-enters the APX Loop. The specificity matters. "Doesn't feel right" is not acceptance feedback. "The tone in section three is too formal for our audience" is.

By the time work reaches final verification, it has been through multiple agent-to-agent iterations and domain-mapped quality gates. The remaining issues should be strategic, not mechanical.

### 9.3 The Reflection as the Link

The reflection is the structural link between Human-Last and Human-First. Without it, APX is a pipeline. With it, APX is an evolving system. This distinction matters more than anything else in the framework.

During the reflection, agents report their APX Metrics. Iteration counts per task. First-pass acceptance rates. Which specs produced clean output and which required heavy iteration. Humans review the data and make decisions. If content tasks consistently require more iterations than code tasks, that's a signal to invest in better Intent Architecture for content. If an agent consistently fails a specific quality check, that's a signal to reconfigure its context. Calibration Impact tells you whether last cycle's changes actually made a difference.

### 9.4 Feedback to Human-First

The decisions made in the reflection become changes to the Human-First configuration. Specs get more detailed in areas where ambiguity caused extra iterations. Workflow routing changes where bottlenecks appeared. Permission maps tighten or loosen based on what agents actually needed. Agent identity files evolve as you learn what context produces the best output.

This is the APX Cycle completing. Human-First designs the system. Agentic-First executes within it. Human-Last verifies and collects data. Calibration feeds back into Human-First, and the next cycle runs on a better system. The humans working in this loop aren't doing the same job cycle after cycle. They're doing a harder, more strategic job each time because the routine work is handled and the remaining decisions require real judgment.

## Chapter 10: Framework Pluggability

*APX wraps around your existing methodology. It doesn't compete with it.*

![APX Pluggability](/images/posts/apx-pluggability.svg)

One of the most common questions I get about APX is whether it replaces existing workflows. It doesn't. APX is designed to wrap around your existing processes, not replace them.

The inner loop of APX, the Agentic-First execution phase, is workflow-agnostic. Whatever methodology your team already uses for organizing and executing work, that methodology becomes the inner loop. APX's outer loop, the Human-First design, Human-Last verification, and Calibration cadence, layers on top.

**Scrum + APX.** If your team runs Scrum, your work cycles are still time-boxed iterations with planning, execution, and review. APX adds the outer structure: Human-First configuration feeds into those cycles, Human-Last verification closes them, and Calibration evolves the system between them. Your Scrum ceremonies still happen. They run inside a larger loop that governs how humans and agents work together.

**Kanban + APX.** If your team uses Kanban, continuous flow is still continuous flow. APX adds verification checkpoints and reflection cadences as a quality layer. The Kanban board tracks the work. APX tracks how well the human-agent system is functioning.

**SAFe + APX.** If your organization runs SAFe or another scaled framework, APX's five domains map naturally to ownership structures within those frameworks. Domain owners align with capability teams or value streams. The calibration cadence aligns with inspect-and-adapt events. The frameworks coexist because they operate at different levels.

**Custom workflows.** Custom workflow that doesn't fit any standard methodology? APX still works. The three phases and the calibration loop don't prescribe how you organize individual tasks. They prescribe the relationship between humans and agents.

The practical implication: you don't need organizational buy-in to "switch to APX." You need buy-in to add Human-First configuration, Human-Last verification, and Calibration around whatever you're already doing. That's a much easier conversation than replacing an established methodology. APX wraps around. It doesn't compete.

## Chapter 11: Anti-Patterns

*Seven failure modes I've seen repeatedly. Each one has a specific fix.*

![Seven Anti-Patterns](/images/posts/apx-antipatterns.svg)

**Anti-Pattern 1: The Everybody Reviews Everything Trap.** When every stakeholder reviews every piece of output regardless of expertise, you get inconsistent feedback and slow cycles. The fix is domain-mapped ownership. The person with the right expertise reviews the right thing.

**Anti-Pattern 2: The Set-and-Forget System.** You configure your agents once and never run reflections. Requirements drift, agent capabilities change, and the system degrades silently. The fix is calibrating every work cycle. No exceptions.

**Anti-Pattern 3: The Promptless Agent.** An agent with no context documents, no brand guidelines, no technical constraints. It produces generic output that needs heavy human editing, which defeats the purpose. The fix is investing in Intent Architecture upfront.

**Anti-Pattern 4: The Unlimited Agent.** An agent with full access to everything: production systems, financial data, external communications. One hallucination away from a real incident. The fix is least privilege with an explicit Permission Map.

**Anti-Pattern 5: The Human-in-the-Middle.** A human reviewing every single iteration between agents. This kills the speed advantage entirely. You end up with an expensive, slow pipeline where a human is the bottleneck at every step. The fix is letting agents review agents first and only escalating to humans for final verification.

**Anti-Pattern 6: The Missing Harness Decision.** Choosing a runtime by default or accident rather than deliberately. You end up fighting the constraints of a system that was never designed for your use case. The fix is making harness selection the first decision, documented in a Harness Decision Record.

**Anti-Pattern 7: The Feeling-Based Reflection.** Running reflections based on how people feel rather than what the data shows. "I think it's going well" is not a calibration signal. The fix is data-driven reflections where agents report metrics and humans make decisions based on those metrics.
