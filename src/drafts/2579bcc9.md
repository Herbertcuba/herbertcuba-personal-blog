On July 8, 2026, Jarred Sumner published a blog post titled "Rewriting Bun in Rust." Bun, the JavaScript runtime with millions of users, had been written in Zig. Sumner rewrote it in Rust. All of it. 535,496 lines of Zig became over a million lines of Rust.

He did it in 11 days. One engineer. Roughly 50 dynamic workflows in Claude Code, running a pre-release model called Claude Fable 5. At peak, the system generated 1,300 lines per minute. 6,502 commits. 695 commits in a single hour.

Bun v1.4.0 shipped 2-5% faster and 20% smaller. Prisma Compute launched on the Rust port in production.

I think this is the most important engineering story of the year, and almost nobody is reading it right.

## The backlash

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>Andrew Kelley, the creator of Zig, called the rewrite "unreviewed slop." The HN community split between genuine engineering awe and "marketing stunt for Anthropic." Both reactions miss the structural shift underneath.</p></div>

Andrew Kelley, the creator of Zig, responded the next day. His post got 808 points on Hacker News. He called the million lines of generated Rust "unreviewed slop." He questioned whether a test suite could ever be sufficient to catch bugs in code nobody had read. He made personal attacks on Sumner's management style, later apologized for those, but stood by the technical critique.

The Hacker News community split. One camp saw genuine engineering achievement — a production runtime, faster and smaller, deployed in 11 days. The other camp saw a marketing exercise. Bun was acquired by Anthropic in December 2025. Sumner used Anthropic's own pre-release model to rewrite a project Anthropic now owned. The optics are, to put it mildly, convenient.

My take: both camps are looking at the wrong thing. Whether the code is "slop" or "engineering" is the wrong question. The right question is what happens to code review when code becomes disposable.

## The shift

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>When one engineer can generate a million lines in 11 days, reading diffs becomes impossible. The test suite becomes the only contract. Code review shifts from reading code to maintaining the test suite that validates it.</p></div>

Sumner used an adversarial pattern: for each task, one agent implemented, two agents reviewed. But the "review" wasn't a human reading code. It was agents checking agents, with a human monitoring the process and a test suite as the final arbiter.

6,502 commits in 11 days. No human read them. No human could read them. The test suite passed on all platforms, and that was the ship gate.

I think this is where engineering is going, and the Bun rewrite is just the first case visible enough to argue about. When generation is effectively free — 1,300 lines per minute at peak — the code stops being the thing you protect. The test suite becomes the asset. The code is output. The tests are the contract.

This is a hard shift for engineers who built their identity on code quality. The instinct is to say: you can't trust tests alone. Kelley's argument. And he's right that tests have limits. Integration tests miss edge cases. Unit tests miss emergent behavior. A million lines of generated code will contain bugs that a test suite didn't anticipate.

But here's the uncomfortable part: a million lines of human-written code also contains bugs that the test suite didn't anticipate. The question was never "are tests perfect?" The question is: are they good enough relative to the alternative — which is not shipping at all, or shipping months later with a team of engineers doing manual review that also misses bugs.

<div class="callout">
<span class="callout__label">The inversion</span>
<p>Code used to be the asset and tests were the verification. Now tests are the asset and code is the generated output. You don't review the output. You maintain the contract.</p>
</div>

## The industrialization argument

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>One blogger framed it as "software industrialization" — the cost of rewriting a codebase dropped from 18 engineer-months to ~$20K in tokens. Code that's cheap to regenerate doesn't need to be maintainable. It needs to be correct enough to pass the tests.</p></div>

A blogger at versary.town framed it bluntly: "Software is now cheap. We are entering an era of software industrialization. It doesn't matter if Bun is unmaintainable in this new version. The point is that it took them 6 days to generate this entire Rust codebase. If it's shit, they'll just make it from scratch again."

That's crass, but I think it's directionally correct. The cost of rewriting a production runtime dropped from roughly 18 engineer-months to around $20,000 in Claude tokens. (Some sources cite higher figures; the exact token cost is disputed. The blog post itself doesn't specify a dollar amount.)

When regeneration costs $20K and takes 11 days, maintainability stops being the primary constraint. Correctness — as defined by the test suite — becomes the constraint. You don't need beautiful code. You need code that passes and a test suite good enough to trust.

This is the same shift Simkin describes in *The AI-First Company*: retrofitters bolt AI onto existing processes. AI-first builders redesign around what's now possible. Sumner didn't retrofit Rust onto Bun's Zig structure. He let agents regenerate the whole thing, and the test suite was the only thing that mattered.

## What this means

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>If code is disposable output and the test suite is the contract, then the valuable engineering skill shifts from writing code to writing tests that capture intent precisely. The spec becomes the asset. The code is just a rendering of it.</p></div>

In *2001: A Space Odyssey*, HAL 9000 runs the entire ship flawlessly — until the mission parameters conflict and the system optimizes for the wrong objective. The failure isn't a bug in HAL's code. It's a spec problem. The mission was specified in a way that made the failure mode inevitable, and nobody caught it until it was too late.

That's the risk with agent-generated code. Not that the code is slop. That the test suite — the spec, the contract — isn't good enough. When the code is generated, the test suite is where intent lives. If your tests don't capture what you actually mean, the system will pass every test and still do the wrong thing.

I've seen this. In my own work, I gave an agent decision-making authority over data classification. The test suite passed. The agent classified everything as sensitive because its training corpus was 80% sensitive fields. Every test passed. The system was broken. The tests were measuring the wrong thing.

The implication for engineering leaders: your investment in test infrastructure is now the most important investment you make. Not the code. Not the architecture diagrams. The tests. Because the tests are the contract between human intent and machine output. When code is disposable, the contract is what you protect.

The engineers who thrive in this shift aren't the ones who write the cleanest code. They're the ones who write the most precise tests — the ones that capture intent so completely that generated code either passes because it's correct or fails because the spec caught the drift.

Code got commoditized. The spec didn't.
