I watched an AI build a financial model with a formula error in every future year. The valuation looked clean. Excel didn't complain. A human reviewer would have caught it in thirty seconds.

The model came from a capable agent, working from a solid brief. The output was coherent, well-structured, neatly formatted — it even included sensitivity tables. What it didn't include was correct math: a referencing mistake that compounded silently across every projected period. The DCF showed a number. The number was wrong. Nobody in the loop noticed until three slides had already been built around it.

That's the problem I want to talk about.

---

## The agent is optimizing for the artifact

When you prompt an AI agent to build something — a financial model, a technical spec, a go-to-market deck — it is optimizing for *completion*. It wants to produce something that looks like the thing you asked for. Coherent. Plausible. Internally consistent enough to pass a glance.

This isn't a flaw in the model. It's the design. Generation and verification are different cognitive tasks. A system tuned to produce output is not the same system you'd want auditing that output. The incentive structure is wrong from the start.

The instinctive fix is a better prompt. "Be rigorous." "Double-check your formulas." "Flag any assumptions." I've watched teams iterate through these for months. They reduce the error rate — slightly. They don't solve the problem.

What's actually happening: you're asking one agent to be both the writer and the critic simultaneously. In practice, the writer wins. It almost always does.

---

## What *Blade Runner* understood about verification

In Ridley Scott's *Blade Runner*, the Voight-Kampff machine doesn't ask the subject to confess. It asks oblique questions and measures involuntary physiological responses — things the subject cannot control. The test works precisely because it's adversarial. It's designed to find what the subject is built to conceal.

The detective asking the questions was never inside the process that created the replicant. That separation is load-bearing.

Replace "replicant" with "AI-generated financial model" and the architecture becomes obvious. You cannot use the same generative process to validate what it created. The system that built the artifact has already committed to a worldview — a set of assumptions, a frame, a direction. It will read its own output through that frame. What it produced will look correct to itself.

You need something that was never inside that frame.

---

## The Hostile Reviewer pattern

The **Hostile Reviewer** is a separate agent — a second model, a different prompt, a structurally independent system — whose only function is to break the artifact produced by the first.

Not to improve it. Not to summarize it. To attack it.

This isn't adversarial AI in the academic sense. It's a workflow architecture decision. One model builds, another attacks. The attacking model receives the artifact cold, with no context about the builder's intentions, no prior conversation to anchor it. It comes with a single instruction set: find what's wrong. Find the assumptions that weren't declared. Find the formula referencing the wrong cell. Find the logical leap buried inside the prose. Find the edge case the spec doesn't cover.

The output of the hostile reviewer isn't a rating. It's a structured list of failures. Each one feeds back into the build.

In my experience, teams resist this for two reasons. First: it feels redundant — you already prompted the first agent to be careful. Second: it's slower — two-pass generation costs more time and compute. Both objections are accurate. Neither one survives contact with a decision made on a wrong number.

The first pass optimizes for the artifact. The second pass optimizes for truth.

---

## The four-stage pipeline

In practice, I've converged on a four-stage structure for anything where correctness actually matters:

**Stage 1 — Source Preparation.** Before any generation starts, the input material gets cleaned, chunked, and tagged. Raw data, reference documents, declared assumptions. This stage is mechanical — and it's where most downstream errors originate. Garbage in, coherent-looking garbage out.

**Stage 2 — Structure Specification.** The first agent produces not the artifact but a *schema* for the artifact. The outline of the financial model. The headings of the spec. The skeleton of the deck. No numbers yet, no prose — just structure, surfaced for human review before anything gets built into it. This is the checkpoint most teams skip, and the one they regret.

**Stage 3 — Constrained Creation.** The builder agent generates against the approved structure. Constraints are explicit: what can be assumed, what must be referenced, what format each output must take. The constraints aren't there to slow things down. They're there to give the hostile reviewer something concrete to test against.

**Stage 4 — Hostile Review.** A structurally separate agent receives the output from Stage 3 and a copy of the constraints from Stage 2. It does not receive the original brief, the conversation history, or any framing from the build process. It asks: does the artifact satisfy the specification? Are the formulas correct? Are the assumptions declared or buried? Where does the logic break?

Findings from Stage 4 route back to Stage 3 for revision — or, when the failures are structural, back to Stage 2.

This isn't elegant. It's four passes where instinct says one should suffice. The quality differential is not marginal.

## Why self-review fails at the architecture level

I want to be specific about the mechanism, because the failure mode is subtle.

Language models produce output based on context. When you ask a model to review what it just generated, it does so using the same context window that produced the original — the same conversational history, the same embedded assumptions, the same implicit direction. The review isn't independent. It's a continuation.

There's a useful analogy from software: you don't let the developer who wrote the module write all the tests for it, in the same sitting, with no specification to validate against. Not because developers are untrustworthy. Because they're too close. They'll test the behavior the code *exhibits*, not the behavior the requirements *demanded*.

The hostile reviewer solves this structurally. Different context. Different instructions. No shared prior state.

Some teams build this with a second model — one vendor's output reviewed by another's. Others build it with the same model but with a hard context reset and an adversarial system prompt. Both approaches work. The architecture matters more than the model choice.

---

## The operating model implication

Zooming out: this is a microcosm of a larger pattern in how AI workflows are being assembled right now.

The **AI Operating Model** that actually holds up under load isn't "prompt an agent and ship the output." It's an assembly of agents in deliberate roles — some generative, some critical, some validating, some routing. The hostile reviewer is the critical function. It exists because the generative function cannot fill it.

When you build decision-memo factories — AI systems that produce analysis, recommendations, or structured outputs for human decisions — the hostile reviewer sits between the generator and the decision-maker. It's the layer of trust. Without it, what reaches the decision-maker is an artifact optimized to *appear* correct.

That financial model I mentioned reached a decision-maker. The decision was made on a wrong number. Three months of downstream work was built on top of it.

A second agent, a cold context, a structured list of failures — none of that would have been glamorous. It would have cost an extra pass. It also would have been enough.

The question isn't whether to build the hostile reviewer into your pipeline. The question is how long you're willing to ship without it.
