I used to write documentation for humans who could ask me questions. Now I write it for agents that cannot.

That shift happened quietly. No announcement. No migration guide. One day the primary reader of your runbook, your decision memo, your API spec — that reader became a machine.

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>The primary audience for your documentation has changed. Most teams have not noticed yet.</p></div>

## The standard just changed

Nate B. Jones put it plainly in a video on building AI teams: if you have ambiguous documents, you are just spreading chaos through that system.

Not confusion. Not inefficiency. Chaos. Because an agent does not read ambiguity and ask for clarification. It reads ambiguity and picks one interpretation. Confidently. At scale. Across every task it touches downstream.

The old standard for good documentation was: clear enough that a smart person can follow it. The new standard is: unambiguous enough that a literal executor cannot go wrong.

That is a different bar. And most documentation in most organizations is nowhere near it.

I think about what that means for the things we write every day. A decision memo that says "use your judgment on edge cases" is no longer a reasonable instruction. An API spec that says "handle errors appropriately" is no longer acceptable. A runbook that says "escalate if something seems off" is now a liability.

The documentation that felt good enough last year is actively dangerous today.

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>When agents are the primary reader, "clear enough" is no longer good enough. Ambiguity becomes confident wrong decisions at scale.</p></div>

## Agent-first is now, not later

Here is where I think teams are getting this wrong: they treat agent-readability as something to retrofit later, once they have more agents, once the workflows are more mature.

I think that is backwards.

***Agent-first*** is a design principle for today, not a future state to prepare for. Every document you write today will be read by agents tomorrow. Maybe your agents, maybe someone else's. Maybe a customer's agent parsing your API docs. Maybe a future hire's agent onboarding through your wiki. The document has a longer life than the context in which you wrote it.

My take: if you are not writing documentation with agents as the primary audience right now, you are already accumulating debt. Not technical debt. Clarity debt. And unlike technical debt, you do not always know where it lives until something breaks.

The question to ask before publishing any document: can an agent follow this without asking a single clarifying question? If the answer is no, the document is not finished.

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>Agent-first documentation is not a future upgrade. It is a design principle for today. Every document you write now will be read by machines tomorrow.</p></div>

## Agents as first-class citizens

This goes further than writing style. I think we need to treat agents as ***first-class citizens*** of our documentation — the same way we name human audiences.

Most documentation today is written for someone. "This guide is for engineers onboarding to the payments service." "This memo is for the leadership team." The audience is named. Its context is assumed. Its questions are anticipated.

We should do the same for agents.

"This runbook is read by the incident-response agent. It does not have access to Slack history or call recordings. All context must be in this document." That one sentence changes how you write everything that follows.

When you name the agent as a reader, you stop assuming it knows things it does not. You stop leaving blanks that a human colleague would fill in from memory. You start writing the context explicitly, because you know the reader cannot infer it.

This is not a new discipline. Amazon's six-page narrative culture got here before AI did — the idea that writing a thing down forces you to confront whether you actually understand it. Agent-first documentation is the same forcing function, just with higher stakes. The agent will not nod politely and ask about it later. It will act.

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>Name agents as an audience explicitly in your documentation. When you write for a reader that cannot infer or ask, you are forced to say everything that matters.</p></div>

## What this requires

Writing is no longer a soft skill in the agentic org. It is a hard interface.

The documentation layer is where human intent becomes agent instruction. Get it right, and the agent executes with precision. Get it wrong, and the error propagates through every task it touches, every downstream decision it makes, every escalation path it misreads.

I think the organizations that will pull ahead in the next few years are not the ones with the most agents or the most automation. They are the ones where the humans have gotten serious about the quality of the instructions they give.

That means documentation that states permissions, not just processes. That defines done, not just steps. That names its agent audience, anticipates its context gaps, and leaves no room for confident misinterpretation.

When you write for an agent that cannot ask questions, you find out quickly whether you actually know what you are saying.
