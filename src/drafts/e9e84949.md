I used to write documentation for humans who could ask me questions. Now I write it for agents that cannot.

The shift happened quietly. No announcement. One day the primary reader of your runbook, your decision memo, your API spec became a machine.

## The standard just changed

If your documents are ambiguous, you are spreading chaos through your system. An agent does not read ambiguity and ask for clarification. It picks one interpretation. Confidently. At scale.

The old standard: clear enough that a smart person can follow it. The new standard: unambiguous enough that a literal executor cannot go wrong.

*Human-first:* "If the service is degraded, escalate to the on-call engineer and use your judgment on severity."

*Agent-first:* "If p99 latency exceeds 800ms for two consecutive minutes, page the on-call engineer via PagerDuty with severity P2. Do not escalate to P1 unless error rate also exceeds 5%."

The first version works because humans fill gaps from experience. The second is what an agent needs: explicit thresholds, named systems, no judgment call. Documentation that felt good enough last year is actively dangerous today.

## Agent-first is now, not later

I think teams are getting this wrong. They treat agent-readability as something to retrofit later, once the workflows mature. That is backwards.

Every document you write now will be read by agents. Maybe yours. Maybe a customer's agent parsing your API docs. The document outlives the context you wrote it in.

The infrastructure is already here. The `llms.txt` convention exists because teams realized their docs were being read by machines. MCP tool descriptions are documentation. OpenAPI specs are documentation.

If you are not writing with agents as the primary audience, you are accumulating ***clarity debt***. The audit is simple: count the documents that reference judgment without naming who exercises it. Each one is a liability. The test before publishing anything: can an agent follow this without asking a single clarifying question? If not, it is not finished.

## Agents as first-class citizens

Most documentation names its audience. "This guide is for engineers onboarding to the payments service." We should do the same for agents.

"This runbook is read by the incident-response agent. It does not have access to Slack history. All context must be in this document." One sentence that changes how you write everything that follows.

When you name the agent as a reader, you stop assuming shared context. You stop leaving blanks a colleague would fill from memory. You write the context explicitly, because this reader cannot infer it.

## The tension worth naming

The counterargument is real: optimize entirely for literal executors and you get a different failure mode. A runbook that says "escalate if something seems off" is vague, but a human applies fifteen years of context to it. An agent given explicit rules executes them precisely — including in edge cases the rules were never designed for.

The goal is not to eliminate judgment. It is to eliminate *unintentional* ambiguity. Intentional ambiguity should be named as such: "This step requires human review. The agent should pause and request approval." That is an instruction, not a gap.

## What this requires

Writing is no longer a soft skill in the agentic org. It is a hard interface.

The organizations that pull ahead will not be the ones with the most agents. They will be the ones whose humans got serious about the quality of their instructions. Documentation that states permissions, not just processes. That defines done, not just steps. That distinguishes between "the agent should act" and "the agent should stop and ask."

Documentation is the interface between human intent and machine execution. The interface is the product.
