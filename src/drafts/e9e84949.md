I used to write documentation for humans who could ask me questions. Now I write it for agents that cannot.

That shift happened quietly. No announcement. No migration guide. One day the primary reader of your runbook, your decision memo, your API spec — that reader became a machine.

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>The primary audience for your documentation has changed. Most teams have not noticed yet.</p></div>

## The standard just changed

If you have ambiguous documents, you are spreading chaos through your system.

Not confusion. Not inefficiency. Chaos. Because an agent does not read ambiguity and ask for clarification. It reads ambiguity and picks one interpretation. Confidently. At scale. Across every task it touches downstream.

The old standard for good documentation was: clear enough that a smart person can follow it. The new standard is: unambiguous enough that a literal executor cannot go wrong.

Here is what that looks like in practice. Consider two versions of the same incident runbook line:

*Human-first:* "If the service is degraded, escalate to the on-call engineer and use your judgment on severity."

*Agent-first:* "If p99 latency exceeds 800ms for two consecutive minutes, page the on-call engineer via PagerDuty with severity P2. Do not escalate to P1 unless error rate also exceeds 5%."

The first version works fine when a human reads it — they fill in the gaps from context and experience. The second version is what an agent needs: explicit thresholds, named systems, defined severity tiers. No judgment call required.

The documentation that felt good enough last year is actively dangerous today.

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>When agents are the primary reader, "clear enough" is no longer good enough. Ambiguity becomes confident wrong decisions at scale.</p></div>

## Agent-first is now, not later

I think teams are getting this wrong: they treat agent-readability as something to retrofit later, once the workflows are more mature.

That is backwards.

***Agent-first*** is a design principle for today. Every document you write now will be read by agents. Maybe your agents. Maybe a customer's agent parsing your API docs. Maybe a future hire's agent onboarding through your wiki. The document has a longer life than the context in which you wrote it.

This is also starting to show up in the infrastructure. The `llms.txt` convention — a plain-text file at the root of a website, structured for language model consumption — exists because teams realized their docs were already being read by machines. MCP tool descriptions are documentation. OpenAPI specs are documentation. The machine-readable layer is not coming. It is already here.

If you are not writing with agents as the primary audience right now, you are accumulating ***clarity debt***. Unlike technical debt, you do not always know where it lives until something breaks. The audit is rough but real: count the documents that reference judgment or discretion without naming who exercises it. Each one is a liability.

The question to ask before publishing any document: can an agent follow this without asking a single clarifying question? If the answer is no, the document is not finished.

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>Agent-first documentation is not a future upgrade. It is a design principle for today — and the infrastructure already reflects it.</p></div>

## Agents as first-class citizens

This goes further than writing style. I think we need to treat agents as ***first-class citizens*** of our documentation — the same way we name human audiences.

Most documentation today is written for someone. "This guide is for engineers onboarding to the payments service." "This memo is for the leadership team." The audience is named. Its context is assumed. Its questions are anticipated.

We should do the same for agents.

"This runbook is read by the incident-response agent. It does not have access to Slack history or call recordings. All context must be in this document." That one sentence changes how you write everything that follows.

When you name the agent as a reader, you stop assuming it knows things it does not. You stop leaving blanks that a human colleague would fill in from memory. You start writing the context explicitly, because you know the reader cannot infer it.

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>Name agents as an audience explicitly in your documentation. When you write for a reader that cannot infer or ask, you are forced to say everything that matters.</p></div>

## The tension worth naming

Here is the counterargument: optimize your documentation entirely for literal executors and you risk a different kind of failure.

Human documentation carries implicit flexibility. A runbook that says "escalate if something seems off" is vague, but a human engineer reads that and applies 15 years of context. They know what "off" means in this system, at this time of year, with this traffic pattern. An agent given explicit rules executes them precisely — including in edge cases the rules were never designed for.

Over-specified documentation can be brittle in ways under-specified documentation is not. The rules cover the cases you anticipated. The real world specializes in the ones you did not.

I think this is real, and I think it changes how you write. The goal is not to eliminate judgment. It is to eliminate *unintentional* ambiguity — the kind that comes from assuming your reader shares your context. Intentional ambiguity, where you genuinely want a judgment call, should be named as such. "This step requires human review. The agent should pause and request approval." That is a different instruction than leaving a gap and hoping the agent fills it the way you would.

***Clarity debt*** is not about writing more. It is about writing with awareness of which reader will act on what you wrote.

In the *Roadside Picnic*, the Zone is littered with alien artifacts that follow rules humans do not understand. The stalkers who survive are not the ones who avoided the artifacts. They were the ones who learned which rules each artifact actually followed. The agent is your Zone. You wrote the rules, but you do not fully control how it interprets them. You test against its actual behavior, not against your assumptions about how it should behave. Write the documentation you would write if you knew the reader would follow your instructions exactly as written — including the instructions you did not intend to give.

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>Optimizing for literal executors can make documentation brittle. The goal is not to remove judgment — it is to name intentional judgment calls explicitly, and eliminate unintentional ambiguity.</p></div>

## What this requires

Writing is no longer a soft skill in the agentic org. It is a hard interface.

The organizations that will pull ahead are not the ones with the most agents. They are the ones where the humans have gotten serious about the quality of the instructions they give.

That means documentation that states permissions, not just processes. That defines done, not just steps. That names its agent audience, anticipates its context gaps, and distinguishes between "the agent should act" and "the agent should stop and ask."

When you write for a reader that cannot ask questions, you find out quickly whether you actually know what you are saying.
