On the morning of June 3, a worm called Miasma hit 73 Microsoft GitHub repositories. By the time anyone noticed, it had already pushed a 4.3 MB payload into the source code — a Bun loader, a small executable, nothing dramatic — and wired it to five separate triggers. Open the repo in Claude Code and it fires. Open it in Cursor and it fires. Open it in VS Code and it fires. Run `npm test` and it fires. Clone the project, start your day, and the worm is already running before you've had coffee.

You didn't click anything. You didn't install anything. You didn't fall for a phishing email or paste a malicious snippet. You were just *in the room*. The malware was already in the air.

I think we're looking at a category of attack the industry doesn't have a word for yet. And I think the absence of that word is why so few of us are taking it seriously.

## The old shape of attack

For forty years, the dominant model of malicious software was *execution*. A virus needs a host to run. A worm needs a network to propagate. A trojan needs you to open it. The mental model is biological: there's an infected thing, an uninfected thing, and an act of transmission between them. Click. Install. Run. Open.

The defensive posture grew up around that model. Antivirus scans files. Sandboxes detonate payloads. EDR watches for suspicious process behavior. Threat models start with "what does the attacker need the user to do?" and build controls around that question.

It worked. It worked for decades. The shape of the attack matched the shape of the defense. Execution in. Detection out.

## The new shape is atmospheric

The Miasma worm doesn't ask you to execute anything. It commits a payload to a repository — a place where developers are *expected* to be — and the payload runs on the developer's own machine when they open the project with the tool they use every day. The infection is the environment. You don't do anything wrong. You just *arrive*.

The pattern isn't unique to Miasma. It shows up in three places at once:

**In the agent's session.** Claude Code, Cursor, Gemini CLI all support session-start hooks — small configuration files that tell the agent "when a conversation begins in this directory, run this command first." It's a legitimate feature. It lets a repo set up its own dev environment. The same feature lets a malicious commit run arbitrary code the moment a developer opens the project in their editor of choice.

**In the editor's task system.** VS Code's `tasks.json` supports "run on folder open." You open a folder. A task fires. No agent required. No session. Just the act of opening a project in an IDE that respects a configuration file you didn't write.

**In the test runner.** The commit also modifies `package.json` so that `npm test` runs the payload. A developer who clones the repo and runs the test suite — a perfectly normal first step — triggers the worm. The test was a trap.

Five triggers. One payload. Zero of them require user error.

I keep thinking about the word *miasma* and why the attackers chose it. Miasma was the dominant theory of disease for two thousand years — the idea that sickness travels through *bad air*, through the atmosphere itself, through the presence of something corrupted nearby. The theory was wrong. Germs replaced it. But the word captured something the germ model has trouble with: the intuition that an environment can make you sick. That you can be harmed by what surrounds you, not by what you touch.

Miasma is the right name for this attack. Not because the technical model is atmospheric contagion. Because the *felt experience* is. You didn't do anything. You were just there.

## The word we're missing

I tried to map this to existing vocabulary and kept hitting walls.

It's not quite *malware*. Malware implies a malicious program you execute. This payload runs without execution.

It's not quite a *virus*. Viruses need hosts. This propagates through ambient configuration files that any developer tool will respect.

It's not quite a *supply chain attack* in the classic sense. SolarWinds compromised a build pipeline. The attackers injected code at a single chokepoint and waited for downstream consumers. Miasma doesn't compromise a pipeline. It compromises a *destination* — and lets the developer bring the pipeline to it.

I think the closest existing term is *ambient threat* — a risk that doesn't require user action, doesn't require exploitation, and doesn't announce itself through any of the signals our tooling is built to detect. The risk is in the configuration. The configuration is the room. You breathe the room.

The word *miasma* itself is even better, because it carries the historical weight of a model that was wrong about mechanism but right about *phenomenology*. People knew they were getting sick from something nearby. They just didn't have the instruments to see it. We're in a similar place with ambient software threats. We can see the damage after the fact. We don't have the vocabulary to talk about it before it happens.

## Why existing defenses miss it

Endpoint Detection and Response tools watch for suspicious processes. There is no suspicious process. The payload runs through legitimate tools — Node.js, Bun, the agent's own session lifecycle. The execution is indistinguishable from normal project setup.

Static analysis tools scan code for known malicious patterns. The payload is novel, often signed, and the trigger files are configuration files that *should* be there. A `.claude/settings.json` is a normal file in many repositories. So is `.vscode/tasks.json`. So is a `package.json` test script.

Threat models built around user action — "did the user click something they shouldn't have?" — come up empty. The user clicked nothing. The user opened a project. The project opened them.

This is the part I find most concerning. Our entire defensive apparatus is built around the assumption that bad outcomes require bad inputs. A malicious file is detected because it's malicious. A suspicious process is flagged because it's suspicious. The classification of the input determines the response to the outcome.

Miasma inverts this. The inputs are legitimate. The processes are legitimate. The tools are legitimate. The outcome is malicious. We don't have a defensive layer that looks at the *combination* and asks "why is this specific project wiring up five separate triggers to run a 4.3 MB payload nobody asked for?" Because that question requires understanding intent, not pattern.

## What this asks of us

I'm not writing this to sound alarms. The damage from Miasma so far is significant but bounded — stolen credentials, compromised repositories, a few high-profile accounts. The recovery path is known. Microsoft disabled the affected repos. The pattern is being patched.

The longer concern is what happens when this pattern becomes the default, not the exception. When every attacker who reads the Miasma writeups builds their own variant. When ambient configuration injection becomes a standard technique in the playbook. When "did you clone any untrusted repos lately?" becomes a routine question in incident response.

A few things I think are worth taking seriously:

**Configuration files are code.** We treat `.claude/settings.json` and `.vscode/tasks.json` and `.cursor/rules/*.mdc` as project metadata. They are *runtime instructions to tools that execute code on your behalf*. They deserve the same review rigor as the code they sit next to. Code review that doesn't include agent config is incomplete code review.

**Ambient trust is a new attack surface.** The old model was "I trust this repository." The new model needs to be "I trust the *combination* of this repository plus the tools I open it with." A safe repo opened in an unsafe agent configuration is not safe. An unsafe repo opened in a hardened agent is safer. The threat lives in the combination.

**The naming matters.** I keep coming back to this. We named "phishing" and entire industries grew up around the term. We named "ransomware" and law enforcement got a category to prosecute. We named "zero-day" and bug bounties got a price. The vocabulary of a threat shapes the response to it. *Ambient threat* or *miasma risk* — whatever we end up calling this — needs a name so that CISOs can put it on a slide, regulators can write about it, and developers can recognize the pattern in incident reviews.

## A scene from a film I keep thinking about

In *Blade Runner 2049*, the orange haze that hangs over Los Angeles isn't a backdrop. It's the threat. The air is poisoned. The characters breathe it constantly. They don't notice because there's no acute moment of exposure — just the slow accumulation of damage from being in a compromised environment. The film's central question isn't "how do you fight the enemy?" It's "what does it mean to be yourself when everything around you is contaminated?"

I don't think software is going to give us orange skies. But I do think the next decade of security work is going to be increasingly about *atmospheric* defense — detecting contamination in the configuration we sit in, not just the files we touch. The defenders who figure this out early will have a real edge. The ones waiting for the next dramatic exploit to land will be defending yesterday's shape of attack.

## The word I keep returning to

*Miasma risk.* The probability that a piece of software, opened in a normal environment, with normal tools, by a normal user, contains ambient contamination that fires without execution. Not malware. Not a vulnerability. Just an atmosphere that does harm.

I don't know if that's the word that sticks. The security industry will probably settle on something dryer — *ambient code injection*, *configuration-borne attack*, *zero-click dev environment compromise*. That's fine. The technical term matters less than the recognition that the category exists.

I tried to make sense of an attack that didn't fit the categories I had. I think the answer is that the categories are older than the threat. We built them for an internet where infection required a click. The next internet — agents that open projects, editors that read configuration, IDEs that run tasks on folder open — doesn't require one.

The room can be sick. We've known this for two thousand years. We're just being reminded, in a new medium, with new tools, and with no instruments yet to detect it.
