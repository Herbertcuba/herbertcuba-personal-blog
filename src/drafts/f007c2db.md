A Head of Engineering posted something on LinkedIn recently that I can't stop thinking about. He wasn't hyping AI. He was worried about it.

His point, roughly: AI agents are introducing the idea that everything we built over twenty years to develop software iteratively and incrementally — Agile, Scrum, Continuous Delivery, the whole apparatus — is now useless, because the agent writes the code. He said he'd read countless posts making exactly that claim. And he doubted the people writing them had ever built a real software product, because building something large and complex requires sustained interaction between people across disciplines, not just fast output from one.

I agree with him completely. And I think the concern is bigger than software.

## The Argument That's Going Around

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>The claim has a kernel of truth — generation really did get faster — which is exactly what makes the wrong conclusion so easy to reach.</p></div>

Here's the version of the claim I keep running into. AI writes the code now, so the scaffolding we built to manage the *uncertainty* of writing code — sprints, standups, code review, staged releases, retrospectives — is dead weight. Just describe what you want. The agent builds it. Ship it.

There's a kernel of truth buried in there, which is exactly why it's dangerous. Generation really did get faster. An agent can produce a working feature branch in minutes that would have taken a senior engineer a day. That part isn't hype. That part is just true.

The mistake is assuming that because the hardest visible step got cheap, the whole problem got cheap. That's not how complex systems work. It's never how they've worked.

## What Agile Was Actually For

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>Agile was never a way to write code faster — it was a way to coordinate humans discovering the right thing to build, and that need only sharpened.</p></div>

I think a lot of the "Agile is dead" commentary misunderstands what Agile was solving in the first place. It was never really a methodology for *writing code faster*. Any competent engineer could write code fast. Agile existed to manage something harder: nobody knows exactly what the right thing to build is until they see it built, used, and reacted to. The rituals — sprints, standups, retros — were coordination mechanisms for humans discovering the right shape of a product together, under uncertainty, across roles that don't naturally speak the same language.

Product doesn't think like engineering. Design doesn't think like ops. Legal doesn't think like anyone. Agile was the connective tissue that let those different ways of seeing the problem collide productively instead of destructively.

None of that need went away because an agent can generate a pull request. If anything, it got sharper, because now the bottleneck isn't "can we build this" — it's "are we building the right thing, and does it actually fit with everything else."

<div class="keymsg"><span class="keymsg__label">The reframe</span><p class="keymsg__text">AI didn't remove the need for discipline. It removed the <span class="hl">excuse</span> to skip it — because now the only thing standing between an idea and a shipped product is the quality of your <span class="hl">judgment</span>.</p></div>

## The Constraint Didn't Disappear, It Relocated

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>Speed up one stage of a pipeline and the bottleneck doesn't vanish — it moves to whatever didn't get faster, which right now is review and integration.</p></div>

This is the part I think the "Agile is obsolete" crowd is missing, and it's a pattern I see everywhere AI touches serious work, not just software.

When you make one stage of a pipeline dramatically faster, the bottleneck doesn't vanish. It moves to whatever didn't get faster. In software right now, generation scaled by 10x or more. Review didn't. Integration didn't. The judgment call about whether this feature actually belongs in the product, whether it's secure, whether it fits the architecture three teams are depending on — none of that scaled at all. It's still bounded by human attention, human trust, human accountability.

So the constraint relocated. It used to sit at "can we build this fast enough." Now it sits at "can we decide what to build, sequence it sanely, and integrate it without breaking everything else fast enough." That's a different skill. It's a harder skill. And most organizations haven't built any discipline around it yet, because they never had to — the old constraint was doing the job for them.

## Orchestration Over Execution

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>Execution used to be the scarce thing that captured the value; now that agents make it cheap, orchestration is the scarce thing — and almost no one has built muscle around it.</p></div>

I want to name what I think the actual shift is, because "AI changes software development" is too vague to act on.

The shift is this: value used to concentrate in execution. Whoever could write the cleanest code, the fastest query, the tightest architecture, captured most of the value in the chain. Execution was scarce, so execution got rewarded.

Execution isn't scarce anymore. Orchestration is.

By orchestration I mean the layer of decisions that sits above any single task: what should we build, in what order, with what tradeoffs, reviewed by whom, integrated how, owned by which person when it breaks at 2am. That layer was always there. It just used to be cheap relative to execution, so nobody built serious muscle around it. Now it's the whole game.

<div class="scifi"><span class="scifi__label">Meanwhile, in science fiction</span><p class="scifi__film">Apollo 13 — the room, not the rocket</p><p>The rocket was already built. The engines already worked. What saved the mission wasn't a faster engineer building a faster part — it was a room full of specialists forced to coordinate under pressure, translating across disciplines in real time, deciding <em>what to try next</em> and in <em>what order</em>, with every wrong sequencing choice costing lives. Execution was never the constraint on that flight. Orchestration was the entire mission.</p></div>

I think that's where every team using AI agents right now actually lives, minus the stakes. The code exists. The question is who decides what code should exist, who checks it against everything else that's true about the system, and who's accountable when three different agents produce three technically-correct-but-mutually-incompatible answers. That last failure mode is not hypothetical — it's the most common way agent-heavy teams generate work faster and ship slower, because nobody owns the reconciliation.

## What Orchestration Discipline Actually Looks Like

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>Orchestration isn't a vibe — it's a concrete operating layer with named owners, a review cadence, decision logs, and a few metrics that make judgment visible enough to manage.</p></div>

Naming the shift is the easy part. The harder question — the one a CIO or a Head of Digital actually has to answer on Monday — is what you build so orchestration stops being a word and starts being how the team runs. In my experience it has three moving parts, and none of them are new. They're the old disciplines, rebuilt for a world where generation is instant.

The first is a **decision layer with a name attached**. Every AI-touched surface — a codebase, a customer-facing answer, a contract clause, a campaign — needs a single accountable owner for what ships, not "the AI team" in the abstract. When execution is cheap, the decision to execute is the only expensive thing left, so it's the thing that needs an owner. Map each surface to a person. That map is the backbone.

The second is a **cadence built around three phases, not one**. Teams used to run one loop: build, then review at the end. That collapses when building is instant. The loop that holds up has a *frame* phase (decide what's actually worth building and what "right" means before an agent touches it), an *execute* phase (agents generate, at speed), and a *calibrate* phase (review, integrate, and feed what you learned back into the next frame). The frame and calibrate phases are where the humans now live. Skip either one and you get fast, confident, incoherent output.

The third is **making judgment legible** — a short list of metrics so orchestration can be managed instead of felt. The ones I'd watch:

<div class="spec-doc">
<span class="spec-doc__label">an orchestration starter — what to stand up first</span>
<ul>
<li><b>First-pass acceptance rate</b> — share of agent output that ships without human rework. Rising = your framing is getting sharper. Falling = you're generating faster than you're deciding.</li>
<li><b>Human-touch rate</b> — where a person actually intervened. You want this concentrated on high-stakes surfaces, not sprayed evenly across everything.</li>
<li><b>Integration-conflict rate</b> — how often parallel agent work collides and has to be reconciled. This is the 2am-pager number; it's the truest measure of whether orchestration is working.</li>
<li><b>Decision log</b> — a lightweight record of what was decided, by whom, and why, for every surface that ships. Not bureaucracy — the audit trail that makes the whole thing accountable.</li>
</ul>
</div>

Here's roughly what that looks like when a team actually builds it — a composite, but drawn from real practice, not a benchmark. Before: agents everywhere, generation up maybe 5x, but ship velocity *flat*, because review queues backed up and integration conflicts ate the gains. After a quarter of building the layer above — named owners, the three-phase cadence, the decision log — generation stayed roughly the same, but rework dropped and time-to-ship moved for the first time in months. The point isn't the exact figures, which are directional. It's the shape: the gains were never in generation. They were in the discipline wrapped around it.

<div class="callout"><span class="callout__label">The first move</span><p>If you do one thing this month, do this: pick your three highest-stakes AI surfaces and put a single named owner on each. Everything else — cadence, metrics, logs — is easier to build once someone is accountable for the outcome.</p></div>

And to close the loop every board eventually asks about: this discipline *is* the compliance answer. The named owner, the decision log, the human checkpoint on high-stakes surfaces — that's not overhead bolted onto AI work, it's the same structure a regulator or an auditor asks for. Orchestration discipline and governance turn out to be the same investment, described by two different departments.

## Beyond Software

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>The identical pattern is unfolding in finance, legal, marketing, and ops — cheap generation, an old discipline declared obsolete, and the same wrong conclusion.</p></div>

The reason I think the Head of Engineering's post deserves more attention than a software-industry debate is that the exact same pattern is playing out in every function I've watched adopt AI seriously — finance, legal, marketing, ops. Generation gets cheap. A model can draft the contract clause, the campaign brief, the reconciliation report, the customer response, almost instantly. And in every one of those functions, someone is currently arguing that the old disciplines — review cycles, sign-off chains, audit trails, editorial standards — are now obsolete because "AI does it."

They're wrong for the same reason the "Agile is dead" crowd is wrong. The discipline was never really about the artifact. It was about the judgment that decides whether the artifact is right, safe, and coherent with everything around it. That judgment doesn't get automated by making the artifact faster to produce. It gets more important, because now it's the only human bottleneck left standing between an idea and something real happening in the world.

<div class="callout"><span class="callout__label">The pattern</span><p>Every function tells itself the same story right now: "AI replaced the work." What actually happened is AI replaced the slowest part of the work, and revealed that the fastest part was never the part that mattered most.</p></div>

That's why I keep saying AI is an operating-model shift, not a tool story. A tool story ends at "we generate content faster now." An operating-model shift asks a harder question: who decides, who reviews, who owns the outcome, and how does a team of different disciplines stay coordinated when the thing they're coordinating around can now be produced in seconds instead of days. AI doesn't just change output. It changes how work is organized — and organizations that don't redesign around that are going to ship a lot of fast, technically-correct, mutually incompatible garbage.

## The Discipline Didn't Die

<div class="chapter-tldr"><span class="chapter-tldr__label">In short</span><p>The bottleneck disappeared; the problem didn't — coordinating people who see the world differently around what deserves to be built is now the only problem left.</p></div>

I don't think the Head of Engineering who wrote that post is wrong to be worried. I think he's early. The people declaring Agile and Scrum obsolete because AI writes the code are making the same mistake every generation makes when a new tool arrives: they mistake the disappearance of one bottleneck for the disappearance of the problem.

The problem was never "can we write the code fast enough." It was always "can a group of people who see the world differently agree on what to build, in what order, and hold each other accountable for whether it's right." AI didn't solve that problem. It made it the only problem left.

Agile isn't dying. It's growing up — from a discipline for coordinating people who write code, into a discipline for coordinating people who decide what deserves to be built and built by whom. The work now is to actually build that layer: give it owners, a cadence, and a way to see itself clearly. Get comfortable there. That's where the real work lives now.
